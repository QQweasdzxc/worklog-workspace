# P5.3 Refactoring Sprint 1 — Architecture Review

Build: `20260714-1604`

## 1. Current Data Flow

```text
Knowledge Source
  -> Knowledge Intelligence
  -> Recommendation Candidates
  -> Suggestion Intelligence
       -> Generalize
       -> Match confirmed Work Memory
       -> Deduplicate candidates
  -> AI Suggestion Workspace
  -> User confirms
  -> DataService.saveWorkModelsOnly()
  -> Supabase user_work_models
  -> WorkLog suggestions
```

The new `SuggestionIntelligence` module is pure logic. It does not render UI, mutate Work Memory, call DataService, or access Supabase.

## 4. Technical Debt

### P1 — Cloud ownership of suggestion decisions

`WORK_MEMORY_AI_SUGGESTION_DECISIONS_KEY` is stored in LocalStorage. Ignored and merged suggestions can therefore reappear on another device. Before P5.4 cross-device learning, decisions need a Cloud repository and explicit sync semantics.

### P1 — Work Memory metadata source of truth

Work Model names are synchronized through `user_work_models`, while description, category, source and merge notes still use `zhuge_work_memory_merge_notes_v1`. This is a split source of truth and should be unified before Work Memory becomes a learning engine.

### P1 — Optimistic Work Memory mutation

`persistWorkMemory()` updates local state before Cloud success and does not roll back if `saveWorkModelsOnly()` fails. This conflicts with the stricter WorkLog write-success rule and should be corrected in a dedicated Cloud-flow sprint.

### P2 — `worklog-app.js` remains oversized

The file still owns Workspace rendering, event binding, WorkLog, Knowledge UI, Conversation UI and Work Memory orchestration. This sprint extracted only the low-risk pure Suggestion logic. Feature-module extraction should resume behind browser regression gates.

### P2 — Rule-based generalization vocabulary

Generalization is intentionally conservative and deterministic. Rules need a versioned vocabulary, product review and real anonymized samples before broadening. Over-generalization is a larger trust risk than missing a merge.

### P2 — Duplicate render/bind architecture

Rendering uses large template strings followed by global DOM queries. It is functional but makes component ownership and event lifecycle difficult to reason about. Event delegation or small render components would reduce this cost.

### P2 — Three copied application trees

Root, `apps/web`, and `apps/chrome-extension` contain synchronized copies. A shared source/build pipeline remains the long-term fix, but changing Build Flow is outside this sprint.

### P3 — Legacy local keys and statistics

Merge statistics and legacy decision keys remain for backward compatibility. They should receive a documented retention/migration policy after Cloud decision storage exists.

## 5. Recommended Refactoring Order

1. Cloud-backed Work Memory metadata and suggestion decisions.
2. Make Work Memory persistence transactional: Cloud success, then committed state/render.
3. Extract Work Memory UI/controller from `worklog-app.js`.
4. Extract Knowledge UI/controller without moving Knowledge Intelligence.
5. Introduce shared-source packaging for Web and Extension.
6. Add browser-level regression tests for suggestion adoption and cross-device reload.

## 6. Regression Risk

- Medium: suggestion counts may decrease by design because duplicates and existing Work Memory matches are filtered.
- Medium: conservative thresholds may still require vocabulary tuning with real documents.
- Low: WorkLog CRUD, Conversation, Export, Auth and Supabase schemas were not modified.
- Low: existing decision keys remain recognized to avoid resurfacing previously handled suggestions.
