# P5.6 Work Memory Cloud Foundation — Architecture Review

## Architecture decision

`public.user_work_models` is the only formal source for Mr. KM Work Memory.

```text
Supabase user_work_models
        ↓ load (user_uuid + RLS)
DataService.workModelsState (complete Work Object)
        ↓ projection
workModels() (active names for legacy UI / WorkLog suggestion)
        ↓ cache
LocalCache (offline / performance only)
```

Cloud load replaces the cached Work Memory. Local data is never allowed to overwrite a non-empty Cloud field during hydration.

## Formal sources

| Domain | Formal source |
|---|---|
| Work Memory | `public.user_work_models` |
| WorkLog | `public.work_entries` |
| Conversation | `assistant_conversations`, `assistant_messages`, `assistant_conversation_states` |
| Work Identity | `public.user_work_profiles` |
| Knowledge | `knowledge_sources`, `knowledge_units`, `knowledge_recommendation_candidates` |

## Cache-only data

- `cacheKey("work_models")`: complete Work Object snapshot for offline rendering.
- `profile.tags`: active Work Memory names compatibility projection.
- `zhuge_work_memory_merge_notes_v1`: legacy source, read once for migration and removed after successful Cloud write.

## Work Object

```text
id, user_uuid
name, description, category
aliases
source, source_references
keywords
is_active
familiarity, last_used_at
created_at, updated_at
```

## Migration behavior

1. Render cached Work Memory while Cloud is loading.
2. Load complete Work Objects from `user_work_models`.
3. If legacy metadata exists, fill only Cloud fields that are still empty/default.
4. Save the merged Work Objects through `SupabaseRepository.saveWorkModels()`.
5. Remove the legacy metadata key only after successful Cloud persistence.
6. Replace DataService state and Local Cache with the Cloud response.

The migration is idempotent per browser. A stale browser cannot replace non-empty Cloud description/category/source data.

## RLS and Data API

- RLS remains enabled.
- SELECT/INSERT/UPDATE/DELETE policies require `(select auth.uid()) = user_uuid`.
- UPDATE includes both `USING` and `WITH CHECK`.
- `authenticated` receives explicit table grants because newly created/changed Data API resources may not be exposed automatically.

## Retired formal responsibility

- `zhuge_work_memory_merge_notes_v1` no longer participates in runtime Work Memory reads or edits.
- Name-only `DataService.workModelsState` is retired; state now contains Work Objects.
- `profile.tags` is not a source of truth and is maintained only for older WorkLog controls.

## Remaining Local Storage technical debt

The following are interaction history, not Work Memory fields, and remain local in P5.6:

- `zhuge_work_memory_merge_decisions_v1`
- `zhuge_work_memory_merge_stats_v1`
- `zhuge_work_memory_ai_suggestion_decisions_v1`
- suggestion page position/cache keys

They should move to a user-scoped Cloud feedback/memory table only when P6 requires cross-device learning feedback. They must not be added back into `user_work_models` as unstructured metadata.

## P6 boundary

P6 Work DNA Learning is intentionally excluded. Future work may update familiarity, Work DNA and recommendation reasons from WorkLog/document feedback, but it must use explicit user-confirmed learning events and preserve Work Memory as the authoritative identity of each Work.

## Production verification (2026-07-14)

The migration has been applied to the Supabase project used by `AUTH_CONFIG.supabaseUrl`.

- 16 `user_work_models` columns verified.
- Own-user SELECT / INSERT / UPDATE / DELETE policies verified.
- UPDATE has both `USING` and `WITH CHECK`.
- Authenticated grants verified.
- Primary, user/name unique, user/id unique, active and active/updated indexes verified.
- Existing table contains 51 Work Memory rows across 3 users; existing rows remain intact and receive schema defaults until each user's legacy metadata migration runs.

The application still detects missing P5.6 columns and displays `Work Memory Cloud 尚未初始化` for deployments that have not executed:

- `docs/supabase/20260714_p5_6_work_memory_cloud_foundation_schema.sql`
- `docs/supabase/20260714_p5_6_work_memory_cloud_foundation_verification.sql`

Supabase Advisor also reports security/performance findings on unrelated pre-existing tables, views and functions. They are recorded as project-level technical debt and were not changed in this Work Memory-only sprint.
