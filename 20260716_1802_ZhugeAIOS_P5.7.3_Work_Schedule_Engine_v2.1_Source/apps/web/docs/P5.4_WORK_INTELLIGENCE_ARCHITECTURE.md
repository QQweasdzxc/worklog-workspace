# P5.4 — Work Intelligence Architecture

Build: `20260714-1641`

## Product Boundary

Mr. KM learns work, not documents. Text fragments, units, pages and keywords remain evidence. The product output is a `Work DNA` object that can improve Work Memory and WorkLog suggestions.

## Learning Pipeline

```text
Document
  -> extraction and readable-text quality check
  -> evidence lines with page / section references
  -> WorkIntelligence.discover()
       -> high-level Work Discovery
       -> Work DNA construction
  -> SuggestionIntelligence.prepareCandidates()
       -> compare confirmed Work Memory
       -> deduplicate and generalize legacy candidates
  -> intelligence_summary.works
  -> Learning Review
  -> user confirmation
  -> user_work_models
  -> KM Suggestion
  -> WorkLog
```

## Work DNA v1

```js
{
  id,
  name,
  purpose,
  description,
  processes: [],
  systems: [],
  departments: [],
  outputs: [],
  frequency,
  triggers: [],
  keywords: [],
  evidence: [{ text, pageReference, sectionReference }],
  confidence,
  sourceKnowledgeId,
  version: "work-dna-v1"
}
```

## Storage Decision

P5.4 v1 does not add a table. Work DNA is stored in `knowledge_sources.intelligence_summary.works`; work-level candidates continue through the existing recommendation candidate table. This keeps the Cloud flow stable while the product model is validated.

## Responsibility Boundaries

- `work-intelligence.js`: identify high-level work and build Work DNA. No UI, DataService or Supabase access.
- `suggestion-intelligence.js`: compare Work Memory, generalize legacy action candidates and deduplicate suggestions.
- `knowledge-intelligence.js`: extract document evidence and orchestrate the two pure engines.
- `worklog-app.js`: render Learning Review and collect user confirmation.
- `data-service.js` / repositories: persist existing source, unit and candidate structures.

## Known Limitations

- Work Discovery v1 is a deterministic, versioned ontology. It is strongest for procurement, supplier, payable, ERP, onboarding, training, contract and project workflows.
- Unsupported domains intentionally return no Work instead of turning random action sentences into fake work.
- Work DNA is not yet an independently queryable Cloud entity; promoting it to a formal table requires PM schema approval.
- Learning Review supports confirmation of discovered work. Fine-grained Work DNA editing and “add missing work” are deferred until the underlying Cloud model is confirmed.
- Existing processed documents need “重新學習” before they receive Work DNA v1.
