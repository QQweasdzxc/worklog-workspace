# Zhuge AI OS Product Constitution

## P5 Core Principle

Zhuge AI OS 不是一套功能很多的軟體。

Zhuge AI OS 要培養的是一位會陪伴使用者工作、持續學習、持續成長的 AI 工作夥伴：

> **🪶 Mr. KM**

P5 之後，每一項新功能都必須先回答一個問題：

> **這項功能，是讓 Mr. KM 更了解使用者？還是只是讓系統多一個功能？**

如果只是增加功能，而沒有增加 Mr. KM 對使用者工作的理解能力，就不符合 P5 的方向。

P1 到 P4 的核心，是讓使用者更自然地把工作交給 Mr. KM。

P5 開始，核心提升為：

> **Mr. KM 要怎麼自己成長。**

因此，P5 不只是 Knowledge Intelligence。

P5 更準確的定位是：

> **Learning Intelligence（學習智能）**

Mr. KM 不是在整理文件。

他是在學：

- 工作流程
- 公司制度
- 使用者經驗
- 工作方式

最後形成：

> **Work Capability（工作能力）**

---

## Teach Mr. KM, Not Manage Files

P5 的 Knowledge 不應被設計成文件管理系統、Metadata 後台或資料庫管理介面。

Knowledge 的真正目的，是讓使用者：

> **教 Mr. KM 一項新的工作能力。**

因此，每新增一份文件，不代表系統多了一份檔案，而代表：

> **Mr. KM 又學會了一項工作能力。**

例如，使用者上傳：

```text
採購 SOP
```

產品語言不應是：

```text
新增了一份 PDF
```

而應是：

```text
🪶 Mr. KM 已學會：

✔ 採購流程
✔ 詢價
✔ 驗收
✔ 請款
```

---

## User Responsibility

使用者永遠只需要做兩件事：

1. 提供資訊，例如自然語言、文件、需求。
2. 確認 Mr. KM 是否理解正確。

除此之外，應優先由 Mr. KM 完成：

- 分類
- 標籤
- Metadata
- 規則整理
- 工作拆解
- 建議工作

使用者負責提供內容。

Mr. KM 負責理解內容。

---

## Unified Interaction Loop

整個 Zhuge AI OS 應維持同一個互動模型：

```text
使用者提供資訊
        │
        ▼
🪶 Mr. KM 理解
        │
        ▼
請使用者確認
        │
        ▼
🪶 Mr. KM 學會
        │
        ▼
未來主動協助
```

這個循環適用於：

- 工時
- 知識
- Calendar
- 任務
- 投資
- 旅遊
- 未來所有 Agent 能力

---

## Product Language

P5 之後請避免讓一般使用者看到工程術語。

| Avoid | Prefer |
| --- | --- |
| Knowledge Repository | 藏書閣 |
| 新增文件 | 教 Mr. KM 新知識 |
| Metadata | Mr. KM 自動理解的資料 |
| Knowledge Units | 工作知識 |
| Recommendation Candidates | 可建立工作 |
| Processing | Mr. KM 正在整理 |
| Processed | Mr. KM 已學會 |

---

## Knowledge Card Mental Model

知識卡不應是「文件卡」。

知識卡應是「能力卡」。

它應強調：

```text
Mr. KM 已學會什麼？

未來可以協助使用者做什麼？
```

而不是優先強調：

- PDF
- DOCX
- XLSX
- Version
- Metadata
- Database fields

---

## Product Tagline

> **每一次互動，都讓 Mr. KM 更懂你的工作。**

> **Every interaction helps Mr. KM understand your work better.**

這句話是 Zhuge AI OS P5 之後的產品判斷標準。

不是：

```text
我今天新增了一筆工時。
```

而是：

```text
Mr. KM 今天又更了解我的工作。
```

不是：

```text
我今天上傳了一份 SOP。
```

而是：

```text
Mr. KM 今天又學會了一項新的工作能力。
```

---

## Principle 6: Learning Never Stops

Mr. KM 並不是一次性設定完成的系統。

他是一位會隨著每一次互動、每一份文件、每一次確認，而持續學習與成長的 AI 工作夥伴。

因此：

- 每一句自然語言
- 每一份文件
- 每一次修正
- 每一次確認

都應該讓 Mr. KM 對使用者的工作理解更完整、更精準。

Knowledge Repository 不是知識的終點，而是：

> **Mr. KM 學習歷程的起點。**

---

## Growth Record

P5 之後，產品可以逐步呈現 Mr. KM 的成長，而不是只呈現資料數量。

例如：

```text
🪶 Mr. KM 成長紀錄

今天學會：

✔ 採購工作提醒事項

本週新增：

+3 項工作能力

目前已學會：

42 項工作能力
```

這個設計方向的目的，是讓使用者感受到：

```text
我不是在養資料庫。

我是在培養一位助手。
```

---

## P5 Review Checklist

進入 P5 之後，每次新增功能前都應檢查：

- 這項功能是否讓 Mr. KM 更了解使用者？
- 這項功能是否符合「使用者提供資訊，Mr. KM 負責理解」？
- 這項功能是否避免要求使用者先整理 Metadata？
- 這項功能是否能讓 Mr. KM 未來更主動協助？
- 這項功能是否延續同一個互動模型？
- 這項功能是否能累積 Mr. KM 的工作能力？
- 這項功能是否讓操作更自然，而不是讓功能更多？

如果答案是否定的，請重新思考設計。
